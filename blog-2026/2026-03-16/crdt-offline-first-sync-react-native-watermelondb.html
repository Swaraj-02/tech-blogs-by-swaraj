Technical introduction

Mobile applications that must remain usable without a network connection face a unique set of engineering constraints. The UI thread cannot be blocked by long‑running I/O, the JavaScript runtime must stay responsive while the underlying SQLite store is being queried, and background services need to reconcile divergent state once connectivity returns. At scale, these constraints multiply: dozens of active devices may modify the same logical entity concurrently, and the sync layer must guarantee eventual consistency without imposing prohibitive latency on the user.

React Native, combined with WatermelonDB's JSI‑backed SQLite engine, provides a performant foundation for large state trees. However, WatermelonDB alone does not resolve conflicts that arise when multiple replicas edit the same record offline. Conflict‑free Replicated Data Types (CRDTs) fill that gap by encoding merge semantics directly into the data model, allowing deterministic resolution without a central authority. This article walks through a production‑grade architecture that layers a CRDT synchronization service on top of WatermelonDB, explores performance‑critical implementation details, and documents the pitfalls that surface in real deployments.

Core concept: fundamentals of CRDTs and their suitability for offline‑first mobile apps

CRDTs are data structures that guarantee convergence when replicas apply operations in any order. Two families dominate mobile use cases:

  • State‑based (convergent) CRDTs where each replica periodically sends its full state vector; the merge function is associative, commutative, and idempotent.
  • Operation‑based (commutative) CRDTs where only the intent (e.g., "add item X") is transmitted; receivers apply operations in causal order.

For offline‑first React Native apps, state‑based CRDTs are often simpler because they map cleanly onto WatermelonDB's immutable record snapshots. A typical pattern is to augment each domain entity with a version vector (Lamport clock per device) and a tombstone flag for deletions. When two versions of the same record arrive, the merge algorithm selects the highest clock per device, resolves ties using a deterministic tie‑breaker (e.g., device UUID), and applies field‑level resolution rules (last‑write‑wins for scalar fields, set‑union for collections).

The key advantage is that conflict resolution does not require a round‑trip to a server. Devices can merge locally, providing instant UI feedback, while the server merely acts as a passive relay that stores the latest merged state.

Architecture pattern: integrating a CRDT layer with WatermelonDB and a custom sync service

The architecture consists of three primary layers:

  1. UI / Business Logic – React components dispatch actions through a Redux‑style store. Actions are translated into WatermelonDB mutations via a thin façade.
  2. CRDT Engine – Wraps WatermelonDB models, injects version vectors, and provides applyRemoteChanges and generateLocalOps APIs.
  3. Sync Service – Runs in a background task (React Native Headless JS on Android, Background Fetch on iOS). It pulls batched changes from the server over HTTPS, feeds them to the CRDT engine, and pushes locally generated ops.

All communication between the CRDT engine and WatermelonDB occurs on the JSI thread, eliminating the bridge overhead that would otherwise throttle mutation throughput.

Key components

  • Model extensions – Each WatermelonDB model inherits from CRDTModel which adds deviceId, lamportClock, and deleted fields.
  • Version vector store – A lightweight SQLite table (crdt_clock) holds the latest clock per device for each collection, enabling fast conflict detection.
  • Sync queue – A persisted FIFO queue (sync_outbox) stores serialized ops generated while offline; the background task drains this queue when network is available.

Implementation deep dive: data model design, JSI‑accelerated mutation handling, and bidirectional sync protocol

Data model design

Consider a Task entity used in a collaborative todo app. The schema includes CRDT metadata and domain fields:

{
    name: "tasks",
    columns: [
        {name: "id", type: "string", isPrimary: true},
        {name: "title", type: "string"},
        {name: "completed", type: "boolean"},
        {name: "device_id", type: "string"},
        {name: "lamport_clock", type: "number"},
        {name: "deleted", type: "boolean", default: false}
    ]
}

The device_id and lamport_clock together form a per‑record version vector entry. When a user toggles completed, the mutation increments the local Lamport clock and writes the new value.

JSI‑accelerated mutation handling

WatermelonDB exposes a database.action wrapper that runs on the JSI thread. By keeping CRDT logic inside this wrapper, we avoid costly JSON serialization between JS and native code.

const toggleTask = async (taskId) => {
    await database.action(async () => {
        const task = await database.get('tasks').find(taskId);
        const newClock = task.lamport_clock + 1;
        await task.update(record => {
            record.completed = !record.completed;
            record.lamport_clock = newClock;
            record.device_id = DEVICE_UUID;
        });
        // Enqueue a CRDT op for later sync
        await syncQueue.enqueue({
            type: "update",
            collection: "tasks",
            id: taskId,
            changes: {completed: record.completed},
            lamport_clock: newClock,
            device_id: DEVICE_UUID
        });
    });
};

Note the absence of any await that would cross the JS bridge; the entire block executes natively.

Bidirectional sync protocol

The protocol is deliberately simple: the client sends a JSON payload containing an array of ops, the server replies with a lastSyncToken and an array of remote ops that the client has not yet seen.

POST /sync
{
    "deviceId": "<uuid>",
    "lastSyncToken": "<token>",
    "outbox": [ ... ]
}

The server stores ops in a crdt_ops table indexed by device_id and lamport_clock. When generating the response, it selects all ops with a lamport_clock greater than the client’s token, orders them by clock, and returns them.

On receipt, the client iterates over remote ops, calls crdtEngine.applyRemoteChanges(op), which performs a merge against the local record. After successful merge, the client updates its lastSyncToken to the highest clock seen.

Code example: applying a remote update

async function applyRemoteUpdate(op) {
    await database.action(async () => {
        const collection = database.get(op.collection);
        const existing = await collection.find(op.id).catch(() => null);
        if (!existing) {
            // Insert new record with metadata
            await collection.create(record => {
                record.id = op.id;
                record.title = op.changes.title || "";
                record.completed = op.changes.completed || false;
                record.device_id = op.device_id;
                record.lamport_clock = op.lamport_clock;
            });
            return;
        }
        // Resolve conflict using version vectors
        if (op.lamport_clock > existing.lamport_clock) {
            await existing.update(record => {
                if ("title" in op.changes) record.title = op.changes.title;
                if ("completed" in op.changes) record.completed = op.changes.completed;
                record.lamport_clock = op.lamport_clock;
                record.device_id = op.device_id;
            });
        }
    });
}

All database interactions stay on the JSI thread, guaranteeing sub‑millisecond latency even when processing hundreds of ops in a single sync batch.

Performance/scaling analysis: latency budgeting, bandwidth optimization, and handling millions of records across dozens of devices

Latency budgeting

Phase Target latency (ms) Reason
Local mutation <5 UI must feel instantaneous; JSI ensures no bridge hop.
Queue persistence <2 SQLite write‑ahead log keeps enqueue fast.
Sync upload <200 (Wi‑Fi) Network round‑trip dominates; batch ops to amortize overhead.
Remote merge on server <100 Server processes ops in bulk; indexes on lamport_clock keep it O(log n).
Sync download & apply <300 Apply ops on device using JSI; parallelize per‑collection.

Bandwidth optimization

  • Delta compression – Instead of sending the full record, only the changed fields and their CRDT metadata are transmitted.
  • Op coalescing – Consecutive updates to the same record on the same device are collapsed into a single op with the highest clock before upload.
  • Binary protobuf payload – Switching from JSON to protobuf reduces payload size by ~40 % and speeds parsing on both ends.

Scaling to millions of records

WatermelonDB’s lazy loading and raw query API allow us to keep the in‑memory footprint under 20 % of total rows. A typical production device holds ~2 M task rows; queries that fetch a user’s active tasks use indexed completed and deleted columns, returning <200 rows per screen.

On the server, a sharded PostgreSQL cluster stores the crdt_ops table. Partitioning by collection and device_id ensures that a sync request touching a single collection scans at most a few thousand rows, regardless of global record count.

Concurrency control

Because the CRDT merge is deterministic, we do not need pessimistic locks. However, we must guard against merge storms where a single device generates thousands of ops in a short window (e.g., bulk import). The sync service enforces a configurable rate limit (default 500 ops per minute) and falls back to a bulk import endpoint that writes directly to the server’s master state, then notifies clients via push.

Common failure mode: merge storms, clock drift, schema migration conflicts, and strategies to detect and recover

Merge storms

Symptoms – Sync latency spikes, server CPU spikes, out‑of‑memory errors on device.

Detection – Monitor sync_outbox size; if it exceeds a threshold (e.g., 10 k entries) raise an alert.

Recovery – Pause background sync, compress the outbox into a single bulk op, and resume. Clients can also request a server‑side “snapshot” to reset their local state.

Clock drift

Lamport clocks rely on monotonically increasing counters per device. A buggy client that resets its counter after a crash can cause its ops to be silently ignored.

Mitigation – Store the last used clock in persistent storage (AsyncStorage or native keychain) and never decrement. On app launch, compare persisted clock with the maximum seen in the database; if the persisted value is lower, bump it to max+1.

Schema migration conflicts

When a new field is added to a CRDT‑enabled model, older clients will generate ops lacking that field, and newer clients may interpret missing fields as null rather than “unchanged”.

Strategy – Version the schema (schema_version column) and include it in the op payload. The server applies migration functions that fill default values for missing fields before merging.

Data corruption detection

Periodically run a checksum audit: the server computes a hash of the canonical state for each collection and compares it with a client‑generated hash sent during sync. Mismatches trigger a forced resync where the client discards its local DB and re‑hydrates from a server snapshot.

Key Takeaways

  • CRDTs provide deterministic conflict resolution without a central coordinator, making them ideal for offline‑first React Native apps.
  • Placing CRDT logic on WatermelonDB’s JSI thread eliminates bridge latency and keeps UI interactions snappy.
  • A lightweight sync service that batches ops, coalesces updates, and uses version vectors scales to millions of records and dozens of concurrent devices.
  • Production systems must guard against merge storms, clock drift, and schema migrations; proactive monitoring and versioned migrations prevent silent data loss.
  • Regular checksum audits and outbox size alerts are essential for early detection of corruption and performance regressions.

By adhering to these architectural principles and operational safeguards, teams can deliver robust, real‑time collaborative experiences on mobile devices that remain functional even in the most unreliable network conditions.
